## 2. 문제 정의 및 배경 (Problem & Context)

**어떤 문제를 해결하나요?**

재건축 사업은 약 10년의 장기 사업이며, 분담금이 사업 진행에 따라 계속 변동한다. 특히 **일반분양가**와 **공사비**가 변하면 비례율이 바뀌고, 비례율이 바뀌면 분담금이 바뀌는 구조. 그런데 조합원·매수 검토자가 이 변동성을 직관적으로 시뮬레이션할 수 있는 도구가 시장에 없다.

또한 분담금 부담을 안고 보유할지, 지금 팔지(매도) 판단할 때 필요한 "분담금 포함 총투자금 vs 신축 완공 후 예상가액" 비교도 일반인은 어렵다.

**누구에게 영향을 미치나요?**
- 재건축 단지 소유자 (조합원)
- 재건축 단지 매수 검토자
- 부동산 의사결정 시 정보 비대칭에서 손해 보지 않으려는 일반 사용자

**이 문제를 어떻게 인지했나요?**
- 17년 감정평가 실무에서 반복적으로 받는 질문 패턴
- 시장에 추정분담금은 공시되나, 사업비/일반분양가 변동에 따른 동적 시뮬레이션 도구는 부재
- 위 사실에 기반한 가설 — 정량 사용자 인터뷰는 출시 후 피드백 폼으로 수집

**만약 이 기능을 만들지 않으면?**
- K-Startup 데모/평가 차질
- 시장에서 "분담금 변동성"이 계속 블랙박스 → 사업 핵심 가치 제안 약화
- 후속 유료 모델(재초환 시뮬, 단지별 정밀 시뮬) 진입 차단

## 3. 제안 솔루션 (Proposed Solution)

**개략 설명:**

1. **첫 화면 — 랭킹 리스트:** 5개 단지 + 2개 blurred(예정) 단지. 각 단지의 종전자산 최소 타입 기준 ROI 내림차순.
2. **평형 선택:** 선택된 단지의 종전자산 타입 카드 그리드. 압구정3구역만 차수(AA) 그룹핑.
3. **질문 선택:** 두 질문 카드 — "나 결국 얼마 더 내야 하는 거야?" / "지금 팔아야 하나?"
4. **면책 동의 게이트:** 4섹션 본문 + "동의하고 계속합니다" 버튼.
5. **시뮬레이션 화면 (Q1):** 상단 분담금 결과 5종 + 하단 슬라이더 3종 (T·L·J).
6. **시뮬레이션 화면 (Q2):** 상단 분담금 결과 + 투자수익 결과(KB시세·총투자금·신축예상·마진·ROI) + 하단 슬라이더 4종 (T·L·J·AT).

모든 화면 하단에 "반박하는 당신이 옳습니다" 피드백 코너 ("고쳐주세요" / "우리 단지도 해주세요").

**왜 이 방식인가?**
- 슬라이더 = 변수의 변동 가능성 자체를 사용자가 체감 → "추정치"임을 자연스럽게 인식. 정적 표 출력보다 본질에 부합.
- **한 화면 동시 가시** = 이전 MVP의 핵심 UX 결함(스크롤하며 만져야 했음) 해소. 모바일 CSS Grid 상하 1:1 / PC 좌우 1:1 분할.
- 멘토 서사 = "전문가가 알려준다" 톤. 17년 경력에 부합하는 절제된 권위. Wise·NerdWallet 톤과 동일 계열.

**검토한 대안:**
- 단순 결과 카드 출력 (no 슬라이더): 변동성을 못 보여줌 → 기각
- 다이얼 vs 슬라이더: PC·모바일 양쪽에서 슬라이더가 직관적 → 슬라이더 확정

## 4. 사용자 스토리 (User Stories)

1. **단지 소유자로서**, 평당 공사비가 1,500만원까지 오르면 내 분담금이 얼마가 되는지 슬라이더로 확인하고 싶다, 왜냐하면 **매도 결정에 직접 영향을 주기 때문이다.**

2. **매수 검토자로서**, 현재 KB시세 + 예상 분담금을 합친 총투자금이 신축 완공 후 예상가액보다 낮은지 보고 싶다, 왜냐하면 **투자 ROI를 알아야 결정할 수 있기 때문이다.**

3. **단지 소유자로서**, 우리 단지의 추정 수치가 다른 단지 대비 어디쯤인지 슬라이더 위 마커로 보고 싶다, 왜냐하면 **상대적 위치가 시장 감각을 준다.**

4. **모든 사용자로서**, 추정치에 동의하지 않을 때 "고쳐주세요" / "우리 단지도 해주세요"를 통해 의견을 남기고 싶다, 왜냐하면 **이 서비스가 위키피디아처럼 진화하길 바라기 때문이다.**

5. **첫 방문자로서**, 면책조항을 읽고 명시적으로 동의한 후 시뮬레이션에 진입하고 싶다, 왜냐하면 **이 결과가 추정치임을 분명히 알고 의사결정에 활용하고 싶기 때문이다.**

## 5. 범위 (Scope)

### 필수 포함 (Must-have)

- [ ] **화면 6종 흐름** — 랭킹 → 평형 선택 → 질문 선택 → 면책 동의 → 시뮬레이션(Q1 or Q2)
- [ ] **계산 엔진** — §8의 선형 스케일링 공식 (factor=1에서 엑셀 BB값과 소수 둘째자리까지 일치)
- [ ] **한 화면 동시 가시** — 결과와 슬라이더가 동시에 viewport에 보임 (모바일 상하 / PC 좌우)
- [ ] **5단지 데이터** — 압구정3구역, 신반포2차, 목동6단지, 여의도시범, 대치 은마 (DB_(0527) 행 21~65)
- [ ] **2개 blurred 단지** — 서빙고 신동아, 서초 진흥 (위치만 회색 점, 단지명·ROI 시각적 블러)
- [ ] **동적 랭킹** — 단지별 종전자산(AC) 최소 타입의 BB(수익률) 기준 내림차순 (현재 결과: 압구정3구역 > 신반포2차 > 여의도시범 > 대치은마 > 목동6단지)
- [ ] **압구정3구역 AA 그룹핑** — 1,2차 / 3차 / 4차 / 5차 / 6,7차 / 10차 / 13차 / 14차 / 대림아크로빌 / 대림빌라트 / 현대빌라트. 향후 다른 통합 단지 추가 시 동일 패턴
- [ ] **단지별 AB 표기 규칙** — 압구정 = 정수 + "평" / 신반포·목동·여의도 = "전용 X.XX㎡" / 은마 = AB가 이미 "X평형" 형태 / 압구정 일부(36A·36B) = "X형"
- [ ] **면책 동의 게이트** — 4섹션 본문(스크롤 가능) + "동의하고 계속합니다" 버튼. 세션 내 1회 동의 후 재노출 안 함
- [ ] **피드백 폼** — "고쳐주세요" / "우리 단지도 해주세요" 한 줄 입력 모달
- [ ] **슬라이더 위 다른 단지 마커** — 기준 단지를 제외한 4개 단지의 동일 변수 값을 마커로 표시. 0.6~1.4 범위 밖이면 끝점에 클립
- [ ] **기본값 초기화** / **처음으로** 버튼
- [ ] **분담금 색상 분기** — 환급(음수) = 녹색, 납부(양수) = 주황
- [ ] **마진 색상 분기** — 양수(이익) = 녹색, 음수(손실) = 주황
- [ ] **breadcrumb** — 단지명 / [분담금 시뮬레이션 OR 매도·보유 판단] / 타입명

## 6. 완료 기준 (Acceptance Criteria)

- [ ] **계산 정확도:** 모든 슬라이더 factor=1일 때 출력 비례율·분담금·총투자금·마진·ROI가 DB_(0527) 시트의 AJ·AZ·AU·AV·BB와 소수점 둘째자리까지 일치
- [ ] **한 화면 동시 가시:** iPhone 13 (390×844), Galaxy S22 (360×800), 1280×720 PC에서 슬라이더 조작 중 결과 카드와 슬라이더 모두 viewport에 보임 (스크롤 없이 또는 각 패널 내부 스크롤만 허용)
- [ ] **랭킹 정렬:** 단지별 종전자산 최소 타입의 BB 기준 내림차순 — 현재 데이터 기준 5개 단지 순서가 정확함
- [ ] **슬라이더 범위:** 기준값 × 0.6 ~ 1.4 (T·L·J·AT 동일)
- [ ] **슬라이더 마커:** 다른 단지 4개의 마커가 각 슬라이더에 표시. 범위 밖은 끝점 클립
- [ ] **블러 처리:** 서빙고신동아·서초진흥 카드의 단지명·ROI에 CSS blur 적용. 지도(있다면) 위치는 회색 점
- [ ] **피드백 입력:** 두 버튼 클릭 시 한 줄 입력 모달 노출, 제출 시 저장 (저장소는 §8·§10 참조)
- [ ] **면책 게이트:** 시뮬레이션 진입 전 1회 동의 필수, 동일 세션 내 재노출 없음
- [ ] **breadcrumb·"처음으로" 버튼:** 모든 화면에서 일관 표시
- [ ] **숫자 단위:** 화면 표시 단위는 모두 **억원** (내부 데이터는 백만원 단위 유지, 표시 시 ÷100)
- [ ] **폰트:** Noto Serif KR (헤딩) + Pretendard (본문) — Google Fonts 또는 자체 호스팅

### 디자인 톤앤매너 (멘토 서사)

**컨셉:** "간달프·오비완·마스터요다" — 자랑하지 않는 권위. 차분한 신뢰. 격식.

**색조:**

| 용도 | 값 |
|---|---|
| 배경 (양피지) | `#F7F4ED` |
| 카드 표면 | `#FFFFFF` |
| 카드 보조 표면 | `#FBF9F3` |
| 보더 | `#E5E0D5` |
| 보더 강조 | `#C9C2B2` |
| 주 텍스트 | `#1F2937` |
| 보조 텍스트 | `#4B5563` |
| Muted 텍스트 | `#8A8576` |
| 강조 (숫자, CTA) | `#1E3A5F` (deep navy) |
| 강조 보조 | `#345984` |
| 1위 골드 | `#B8860B` |
| 환급 (양호) | `#15803D` (forest green) |
| 납부 / 손실 | `#B45309` (burnt orange) |

**타이포:**
- 헤딩: **Noto Serif KR** (weight 500/600/700) — 무게감 있는 세리프, "오래된 책 페이지" 느낌
- 본문: **Pretendard Variable** (weight 400/500/600/700)
- 숫자: 본문 폰트 + `font-variant-numeric: tabular-nums; lining-nums;` (자릿수 정렬)

**모션:**
- 페이지 전환: 320ms cubic-bezier(0.16, 1, 0.3, 1) fade + 6px slide-up
- 슬라이더 thumb hover: 120ms ease scale 1.1
- 슬라이더 thumb active: scale 1.15, 배경 → navy 채움
- 카드 hover: 180ms ease, 보더 강조 + 1px translateY(-1)

**금지 항목:**
- 이모지
- 색상 그라데이션, glow, 네온 효과
- 화려한 마이크로 애니메이션, 통통 튀는 spring
- 활기찬 청색 (`#3B82F6` 같은 일반 Tailwind blue)

## 8. 기술 고려사항 (Technical Considerations)

### 아키텍처 권고

한신씨 PRD에서 제안한 **Kotlin Multiplatform (KMP)** 방향 유지 기본. 본 MVP는 React + HTML 단일 파일이므로 마이그레이션 시:

- **공유 모듈 (KMP common):**
  - 계산 엔진 (순수 함수)
  - 데이터 모델 (District / Type / SimulationResult)
  - 포맷터 (억원·만원·% 변환)
- **플랫폼별 (KMP target):**
  - Web (Compose Multiplatform for Web 또는 별도 React)
  - iOS / Android (Compose Multiplatform)
- **상태 관리:** 화면 흐름 1개 + slider state 1개 = ViewModel 2개로 충분

### 계산 엔진 (검증됨)

```kotlin
// 입력: District 베이스값 + Type 정보 + 4개 factor + optional AT 오버라이드
// 모든 화폐 단위: 백만원 (표시 시 ÷100 → 억원)

fun calculate(
    base: DistrictBase,   // T, L, J, H, I, K, M, N, P, V, AH, AI, AJ, AT
    type: TypeInfo,       // AC, AD
    factorT: Double,      // 0.6 ~ 1.4
    factorL: Double,      // 0.6 ~ 1.4
    factorJ: Double,      // 0.6 ~ 1.4
    atOverride: Double?   // null이면 base.AT 사용
): SimulationResult {
    val P_new = base.P * factorT
    val K_new = base.K * factorL
    val I_new = base.I * factorJ
    val H_new = I_new + K_new + base.M + base.N
    val ratio = (H_new - P_new) / base.V             // 비례율
    val AH_new = base.AH * factorL                    // 국평 조분
    val AI_new = base.AI * factorJ                    // 국평 일분
    val rights = type.AC * ratio                      // 권리가액
    val burden = AH_new - rights                      // 분담금 (음수=환급)
    val AT_new = atOverride ?: base.AT
    val totalInvest = type.AD + burden                // 총투자금
    val margin = AT_new - totalInvest                 // 마진
    val roi = if (totalInvest > 0) margin / totalInvest else 0.0
    return SimulationResult(...)
}
```

**검증 결과:** 신반포2차 전용 68.91㎡, factor=1에서
- burden = 982.99 백만원 (엑셀 AZ=982.987 ✓)
- totalInvest = 5082.99 (엑셀 AU=5082.987 ✓)
- margin = 2432.71 (엑셀 AV=2432.712 ✓)
- ROI = 47.86% (엑셀 BB=0.4786 ✓)

### 데이터 모델

```kotlin
data class District(
    val id: String,
    val name: String,          // "압구정3구역"
    val short: String,         // "강남구 압구정동"
    val address: String,
    val coord: Coord,
    val notice: Notice,         // {date, name}
    val base: DistrictBase,
    val similar: SimilarComplex,
    val types: List<TypeInfo>
)

data class DistrictBase(
    val T: Double, val L: Double, val J: Double,
    val H: Double, val I: Double, val K: Double,
    val M: Double, val N: Double, val P: Double, val V: Double,
    val AH: Double, val AI: Double, val AJ: Double, val AT: Double
)

data class TypeInfo(
    val Z: Int,
    val AA: String?,            // 차수 (압구정만)
    val AB: String,             // 타입명
    val AC: Double,             // 종전 세대단가
    val AD: Double,             // KB시세
    val AE: String              // 분양면적 라벨
)
```

### 의존성 / 외부 서비스

| 항목 | 현재 MVP | 프로덕션 검토 사항 |
|---|---|---|
| 단지 데이터 | 코드 내장 (정적) | 정적 JSON 파일 또는 경량 DB (단지 추가 빈도 낮음) |
| KB시세 (AD, AN) | 하드코딩 (2026-05-18 기준) | 주간 수동 업데이트 또는 자동 수집 (§10 #2) |
| 지도 | stylized SVG stub | 네이버 / 카카오 / 자체 SVG (§10 #1) |
| 피드백 저장 | 입력 후 컨펌만 | Google Forms / Firestore / 자체 (§10 #3) |
| 분석 | 미연동 | GA4 권장 (§10 #14) |

### 성능 / 확장성

- 단지 데이터 정적이고 작음(< 50KB), 초기에는 in-memory로 충분
- 초기 트래픽: K-Startup 단계 일 1,000 PV 미만 예상
- 응답 속도: 슬라이더 → 결과 갱신 < 50ms (현재 MVP에서 충분히 만족)

### 보안 / 개인정보

- 사용자 식별 데이터 수집 없음 (피드백 한 줄 텍스트만)
- 면책조항이 법적 보호 1차 라인 — 시뮬레이션 부정확성에 대한 명시적 면책
